## r39 关键教训

1. **半成品延续模式终结**：r38 commit message 标的 4 项闭环（BuildVoteUI / _cachedTier / 6 SFX [Legacy] / TIER_THREAT 注释）grep+Read 三验证全部通过，**首次"半成品延续模式"零再现轮**。这意味着 r38 流程"PM 系统级深挖二次验证 + commit message 三条件检查"开始生效。
2. **worktree 基线漂移陷阱**：PM 创建 worktree 时基于 r37 commit 而非 r38 最新 commit，导致 Agent 看到的代码是 r37 状态，误报"r38 已闭环项缺失"。**教训**：PM 创建 worktree 前必须 `git log` 确认 main 最新 commit；或 worktree 创建后立即 `git merge <latest> --ff-only`。
3. **Agent 误报率持续 60%+**：r39 13 GAP 中 8 项是误报（r38 已闭环 + 设计如此），与 r38 60+ 误报澄清一致。Explore 模式快速侦察的固有局限是缺少 grep+Read 二次验证。**未来 audit 流程仍按"5 Agent 并行 + PM 系统级深挖"标准模式做**，不依赖 Agent 的浅层判断。
4. **真 GAP 趋稳信号**：r39 真 GAP 5 项（含 3 闭环 + 2 r39+ 候选），是 r34 1 项后的最低水位。代码层趋于"接近完成"的稳态，剩余多为 doc 同步 + 边界情况 + 协议对称化。
5. **"最终次"定义需重新评估**：用户 r37 起连续 3 轮"最终次"，每轮都还能挖出新 GAP（r37 25 / r38 6 / r39 3），但 GAP 性质从 MAJOR 真 bug 转为 MINOR doc 失真 + UX 优化 + 协议对称化。**应明确告知用户**：r39 起代码层 MAJOR bug 趋零，剩余 audit 主要价值在 doc 同步与边界情况清理，不再有"重大功能未实装"的发现。
6. **协议对称化检查模式**：r31 game_paused/game_resumed 协议对称（pause 必有 resume / start 必有 stop）；r39 expedition send/recall phase 校验对称（send 有 wrong_phase 反馈则 recall 也应有）。**未来 audit 时优先扫描"对称协议但单边校验"模式**。
