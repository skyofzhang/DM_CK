#### 1) GAP-C42-01 [MAJOR] SurvivalRoom.js:301 GM 模式掉线主播转移漏注协议头（r41 grep 模式漏抓盲区）

**位置**：`Server/src/SurvivalRoom.js:296-310`
**诊断**：r35 GAP-E25-08 在 broadcast() 注入；r40 GAP-PM40-01/02 加 `_sendToPlayer` / `_sendToClient` helper；r41 GAP-PM41-01 改 17 处 ws.send + r41 GAP-PM41-02 改 `_sendToRoomCreator` 注入。但 r41 grep 用模式 `ws\.send.*JSON\.stringify` 仅匹配变量名为 `ws` 的位置，**漏抓** SurvivalRoom.js:301 用 `this.roomCreatorWs.send(JSON.stringify(...))` 写法的 GM 模式掉线主播转移路径。Agent C 通过 grep `\.send.*JSON.stringify`（无 `ws\.` 前缀限定）发现该漏点。

**修复**：改用 `this._sendToClient(this.roomCreatorWs, {type, data})` helper 自动注入 roomId/timestamp/tick/serverTime/priority 五字段。

**触发场景**：
- GM 模式房间（`isGMMode=true` 或 `roomCreatorOpenId` 以 `gm_creator_` 前缀开头）
- 主播 ws 断开（移除）+ 房间还有其他客户端
- → 选第一个 client 作为新主播 → 通知 `join_room_confirm { isRoomCreator: true }`

**影响**：之前缺 tick/serverTime/priority 三字段，新主播客户端 priority 队列消费时该消息被错误归入低优先级或乱序处理；GM 调试模式下主播切换时序错乱；正式抖音模式不受影响（不进入此分支）。修复后所有路径走 `_sendToClient` 自动注入。

## r42 关键教训

1. **B01 协议头单播路径覆盖第 11 轮再现 + r41 grep 模式漏抓盲区**：r41 grep 用模式 `ws\.send.*JSON\.stringify` 仅匹配变量名前缀为 `ws` 的位置，漏抓 `this.roomCreatorWs.send` / `client.send` 等带前缀的写法。**教训**：协议头注入闭环验证应用 `\.send\(JSON\.stringify` 全后缀扫描（无前缀限定），剩余必须 ≤ 三个 helper 内部（`_sendToClient` / `_sendToPlayer` / `_sendToRoomCreator`）。r42 已用此模式扫描，确认 0 漏点。

2. **半成品延续模式跨轮稳定 1 轮**：r36→r40 跨 4 轮（4 个月）才发现 _broadcastWeeklyRanking 漏修；r40→r41 跨 1 轮发现 17 处 + 1 helper；**r41→r42 跨 1 轮发现 1 处 GM 模式漏点**。信号：PM 系统级深挖二次验证强度持续保持，每个 commit message"全覆盖"声明都需用 grep 硬验证。grep 模式选择影响发现率 — 越宽（无前缀限定）发现率越高。

3. **Agent 误报率 67% 与 r38-r41 同区间**：Agent 5 路 Explore very thorough 模式仍依赖 PM 系统级深挖二次验证。本轮 Agent C 同样通过 grep 一次发现核心 GAP（GM 模式 SurvivalRoom.js:301），印证 r41 教训"分配 grep 扫描类任务给某 Agent"的有效性。

4. **协议头注入是横切关注点（cross-cutting concern）持续成立**：B01 协议头三字段（tick/serverTime/priority）应在最低层 helper 强制注入。r42 闭环将 GM 模式掉线主播转移路径汇聚到 `_sendToClient` helper，未来任何新增单播都通过三个 helper 入口，避免再次出现"漏注入"问题。但 grep 模式选择仍是关键 — 必须用"全后缀扫描"而非"前缀限定"。

5. **真 GAP 趋稳信号 r39→r40→r41→r42（5/4/5/1）— 代码层 MAJOR 极度趋零**：r42 是 audit 史首次"仅 1 真 MAJOR + 0 doc-only MAJOR"轮，剩余 5 项保留 r43+ 全部是 doc 精确化或 POSSIBLE UX 优化（D42-04 跨界依赖 / E42-01 D1 难度系数描述孤儿 / E42-02 实装边界 / E42-03/E42-04 doc 精确化）。**audit 价值持续在 doc 同步与单播路径覆盖盲区清理**。"最终次"定义需重新评估 — r42 仍能挖出 1 真 MAJOR（来自 r41 commit 标"全覆盖"的 grep 模式盲区），audit 价值持续存在。

6. **grep 模式选择是 audit 关键工具**：r41→r42 教训证明，grep 模式的"前缀限定" vs "全后缀扫描"会导致漏检率不同。未来 audit 应建立"grep 模式标准化清单"（如 `\.send\(JSON\.stringify` / `private *\w+ *=*` 字段定义但永不赋值检测 / `OnClick.*AddListener` UI 入口可达性 / `case "[a-z_]+":` 协议路由 vs Data 类对齐），每轮 audit 用全套模式扫描。
