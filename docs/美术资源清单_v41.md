## r41 关键教训

1. **B01 协议头单播路径覆盖第 10 轮再现 + r40 漏修隐藏点**：r40 commit 标"全覆盖"仅改了显式 helper 化路径（`_sendToPlayer` / `_sendToClient` / `_requireBroadcaster` / `_sendToStateToClient`），但 SurvivalRoom.js 中 17 处直 `ws.send(JSON.stringify(...))` 路径 + SurvivalGameEngine `_sendToRoomCreator` helper 未注入。**教训**：协议头注入闭环必须用 `grep "ws\.send.*JSON\.stringify"` 全文扫描验证，剩余数必须为 0（除 helper 内部）。`_sendToClient` / `_sendToPlayer` / `_sendToRoomCreator` 三个 helper 是"白名单"，其他位置都是绕过点。

2. **半成品延续模式跨轮缩短**：r36→r40 跨 4 轮（4 个月）才发现 _broadcastWeeklyRanking 漏修；r40→r41 跨 1 轮（1 天）就发现 17 处 + 1 helper 漏修。**信号**：PM 系统级深挖二次验证强度持续提高，未来 audit 应保持每个 commit message"全覆盖"声明都用 grep 硬验证，不能仅按显式改动 commit message 描述断定。

3. **Agent 误报率 74% 与 r38/r39/r40 同区间**：Agent 5 路 Explore very thorough 模式仍依赖 PM 系统级深挖二次验证。但本轮 Agent C 通过 grep 一次发现核心 GAP（17 处 ws.send 列表完整），证明 Agent 在"模式识别 + grep 模式"下的发现率高于"逐行细读"。**未来 audit 优化**：分配给某 Agent 专做 grep 扫描类任务（如 ws.send 全文扫描 / case 路由 vs Data 类对齐扫描），其他 Agent 做 doc 失真 + 协议字段双向消费检查。

4. **协议头注入是横切关注点（cross-cutting concern）**：B01 协议头三字段（tick/serverTime/priority）应在最低层 helper 强制注入，而非每个调用点零散处理。r41 闭环将所有单播路径汇聚到 `_sendToClient` / `_sendToPlayer` / `_sendToRoomCreator` 三个 helper，未来任何新增单播都通过这三个入口，避免再次出现"漏注入"问题。

5. **真 GAP 趋稳信号 r39→r40→r41**：r39 真 5 / r40 真 4 / r41 真 5，剩余多为 doc 精确化（E41-01~05）+ POSSIBLE UX 优化（C41-06/07）+ 候选项（B01 客户端 priority 队列 1d+ 工时保留）。代码层 MAJOR bug 已趋零，audit 主要价值在 doc 同步与单播路径覆盖盲区清理。"最终次"定义需重新评估 — r41 仍能挖出 5 真 GAP（其中 2 个 MAJOR 来自 r40 commit 标"全覆盖"的盲区），audit 价值持续存在。
