## r40 关键教训

1. **半成品延续模式第 9 轮再现（r36 → r40 跨 4 轮才被识别）**：r36 commit message 标"主播/周榜走 broadcast()"实际未真闭环（仍用 for-loop client.send）。这是 audit 史最长延续期 — r38 首次实现"零再现轮"后，r39 维持，r40 又被翻出 r36 漏修。**教训**：commit message 三条件检查必须包含"代码层 grep 确认实际改动"，不能仅看 commit message 描述；并且 PM 创建 worktree 前必须 `git log --all --oneline | head -30` 确认 main 最新 commit 是否包含目标修复（r38 worktree 基线漂移陷阱在 r40 也避免了）。
2. **Agent 误报率 82%（高于 r38/r39 的 60%）**：r40 22 GAP 中 18 项是误报或已实装。原因：5 Agent 都用 Explore very thorough 模式，依然受限于"快速侦察 + 缺 grep+Read 二次验证"。**未来 audit 仍按"5 Agent 并行 + PM 系统级深挖"模式做**，PM 必须 grep+Read 验证每个 MAJOR 候选，不能依赖 Agent 浅层判断。
3. **B01 协议头单播路径覆盖缺失模式**：r35 实装 broadcast() 注入但漏掉 _sendToPlayer / _requireBroadcaster / _sendStateToClient 等单播路径，导致 30+ 处单播错误响应缺三字段。**教训**：协议头注入必须覆盖**所有发送出口**（broadcast / unicast / response 三类），不能仅 broadcast 路径。r40 通过 `_injectProtocolHeader` helper 抽取 + `_sendToClient` helper 引入 + `_sendToPlayer` 改造一次性覆盖全部出口。
4. **"零候选"轮首次实现**：r39 5 真 GAP / 3 闭环 + 2 候选；r40 4 真 GAP / **4 闭环 + 0 候选**。这是 audit 流程史首次将真 GAP 当轮全部闭环，意味着代码层从 r37 起经过 r38/r39/r40 三轮深挖后，已无明显遗留 — **下轮 audit 起点应该更高（仅审 r41 当轮新发现，不再有累积候选积压）**。
5. **协议头单播路径全覆盖 = B01 协议头 MVP 100% 完成度**（r35 70% → r36 95% → r40 100%）：r35 标 70%（broadcast 路径注入），r36 标 95%（声称主播/周榜也走 broadcast 但未真闭环），r40 真 100%（_sendToPlayer / _sendToClient / 全部 ws.send 路径都注入）。**剩余仅客户端 priority 队列消费 1d+ 工时项保留 r41+ 候选**（已在 r35 时即明确）。
6. **协议对称化检查模式持续有效**：r31 game_paused/game_resumed / r39 expedition send/recall / r40 protocol header broadcast/unicast — 每轮都通过"对称协议但单边校验/单边注入"模式发现新 GAP。**未来 audit 优先扫描**：A 类对称（动作 → 反馈）、B 类对称（开始 → 结束）、C 类对称（广播 → 单播 → 响应 三路径覆盖）。
