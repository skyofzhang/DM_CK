## 1. 游戏概述

### 1.1 一句话定位

《极地生存法则》是一款**抖音直播间多人协作生存游戏**。观众通过发送弹幕指令控制矿工采集资源、抵御夜晚怪物入侵，共同守护极地堡垒坚持到目标天数，即为胜利。

### 1.2 核心体验

- **白天**：观众发送工作指令（1/2/3/4），矿工分别采集食物、煤炭、矿石、添柴升温，维持堡垒运转
- **夜晚**：怪物从地图边缘涌来，观众发送"6"指挥矿工抵御，礼物可触发特殊战斗技能
- **全程**：任何时候刷礼物都能直接改变游戏局势（修城、补资源、消灭怪物等）

### 1.3 目标受众

- **直播观众**：有参与感的轻度交互玩家，无需持续关注，随时弹幕即可参与
- **主播**：具有策略性的直播内容，玩家越活跃主播收益越高

### 1.4 设计目标

| 目标 | 指标 |
|------|------|
| 观众参与门槛 | 0门槛，发送数字弹幕即可 |
| 游戏内容深度 | ≥1个月不重复（随机事件+昼夜递增难度） |
| 礼物消费阶梯 | 0.1抖币（T1）→ 520抖币（T6），6档次 |
| 目标场均在线 | 10~100人 |

## 2. 核心循环

### 2.1 宏观流程

```
游戏开始（选择难度）
    │
    ▼
Day 1 白天（120s）：采集资源 → 消耗资源 → 积累积分
    │
    ▼
Night 1 夜晚（120s）：怪物涌来（含Boss）→ 矿工防守 → 击杀怪物 → 积分奖励
    │
    ▼
Day 2 / Night 2 / ... 循环
    │
    ├── 任意时刻失败条件满足 → 游戏失败（结算）
    │       · 食物归零            reason="food_depleted"
    │       · 炉温 ≤ -100℃       reason="temp_freeze"
    │       · 城门HP = 0          reason="gate_breached"
    │
    └── 达到目标天数 → 游戏胜利（结算）reason="survived"
            Easy:30天 / Normal:50天 / Hard:70天
    
结算后10秒 → 服务器自动以同难度重启新一局（直播间持续运行）
```

### 2.2 每天循环结构

```
白天开始（_enterDay）
  └─ 所有死亡矿工自动复活（_reviveAllWorkers('day_started')）
  └─ 随机事件计时器重置（90-120s后首次触发）
  └─ 观众弹幕：1=食物 / 2=煤炭 / 3=矿石 / 4=升温（资源+效率倍率）
  └─ 资源持续衰减（食物每秒/炉温无煤时下降）
  └─ 矿石自动修城门（每2s消耗1矿石修复5HP，城门有损耗时）
  └─ 礼物随时可触发
  └─ 每90-120s检查随机事件

夜晚开始（_enterNight）
  └─ 矿工HP初始化为满血（_initWorkerHp）
  └─ 所有矿工前往防守位（城门前）
  └─ Boss出现通知广播（boss_appeared，每晚必出，Day1起）
  └─ 3秒后开始生成怪物波次
  └─ 怪物波次伤害：均摊给所有存活矿工，矿工全死后溢出打城门
  └─ 观众弹幕：6=攻击怪物（攻击优先级：普通>精英>Boss）
  └─ 夜晚积分奖励自动入池（_poolNightBase × 当前天数）
  └─ 礼物随时可触发

夜晚结束条件（满足其一）
  └─ 倒计时耗尽（120s）
  └─ Boss被击杀 → 立即触发 night_cleared { reason: 'boss_killed' }
```

## 3. 游戏状态机

### 3.1 服务器端状态（SurvivalGameEngine）

```
idle → day → night → day → night → ... → settlement
              ↑_______↑（循环直到胜利/失败）
```

| 状态 | 含义 |
|------|------|
| `idle` | 等待开始 |
| `day` | 白天运行中 |
| `night` | 夜晚运行中 |
| `settlement` | 结算中（10秒后自动重置→重新开始） |

### 3.2 Unity客户端状态（SurvivalGameManager）

```
Idle → Connecting → Waiting → Loading → Running → Settlement → Waiting
```

| 状态 | 含义 |
|------|------|
| `Idle` | 初始/断开状态 |
| `Connecting` | 正在连接WebSocket |
| `Waiting` | 已连接，等待主播开始游戏 |
| `Loading` | 场景加载中（超时=15s） |
| `Running` | 游戏运行中（对应服务器day/night） |
| `Settlement` | 结算界面显示中 |

> ⚠️ 客户端 Settlement/Idle 状态下，除 join_room_confirm / weekly_ranking / streamer_ranking 外，所有服务器消息被忽略（防止结算期间礼物等继续触发）

## 4. 昼夜系统

### 4.1 时间配置

| 参数 | Easy | Normal | Hard |
|------|------|--------|------|
| 白天时长 | 120s | 120s | 120s |
| 夜晚时长 | 120s | 120s | 120s |
| 胜利目标天数 | 30天 | 50天 | 70天 |

> **注意**：Unity `DayNightCycleManager` 本地默认值为 `_defaultDayDuration=240f` / `_defaultNightDuration=150f`，这些是服务器推送 `phaseDuration` 前的占位值，**实际时长由服务器 phase_changed 中的 phaseDuration 字段覆盖**（来自 `_applyDifficulty` 设置的 120s）。config/default.json 的 `survivalDayDuration:240` 也是 `_applyDifficulty` 执行前的兜底值，正常运行时不生效。

### 4.2 昼夜切换事件

**白天 → 夜晚**：
- 服务器推送 `phase_changed { phase: "night", day, phaseDuration }`
- 矿工HP初始化为满血（每晚重置）
- 矿工全部前往 _gateDefenseCenter（城门前防守集结点）
- 弹幕指令1-4无意义（白天指令服务器仍接收但只在day/night状态处理），6 有效
- Boss出现通知 `boss_appeared { day, bossHp, bossAtk }` 广播
- 3秒后怪物波次开始

**夜晚 → 白天**：
- 服务器推送 `phase_changed { phase: "day", day, phaseDuration }`
- 所有死亡矿工自动复活
- 矿工恢复待机圈（等待新工作指令）
- 夜晚防守积分奖励入池：`_poolNightBase × currentDay`
- 随机事件计时器重置

## 5. 资源系统

> **架构说明**：Unity `ResourceSystem.cs` 只缓存来自服务器的值（`ApplyServerUpdate`），不运行本地衰减逻辑。所有资源计算均在服务器 `SurvivalGameEngine.js` 中执行，客户端收到 `resource_update`（每5s）后刷新显示。Unity Inspector 中的 `_initFood/Coal/Ore` 初始值只在首帧用于显示，5s 内会被服务器数据覆盖。

### 5.1 五种资源

| 资源 | 作用 | 上限 | 归零后果 |
|------|------|------|---------|
| 食物（Food） | 核心生存资源，持续消耗 | 2000 | 立即触发失败（reason="food_depleted"） |
| 煤炭（Coal） | 维持炉温，自动烧煤 | 1500 | 炉温开始下降 |
| 矿石（Ore） | 自动修复城门 | 800 | 城门无法自修 |
| 炉温（FurnaceTemp） | 极地生存核心指标 | **上限100℃** | ≤ -100℃触发失败（reason="temp_freeze"） |
| 城门HP（GateHp） | 防御核心 | 随等级变化（1000~3000） | = 0触发失败（reason="gate_breached"） |

### 5.2 初始值（按难度）

| 资源 | Easy | Normal | Hard |
|------|------|--------|------|
| 食物 | 160 | 200 | 300 |
| 煤炭 | 96 | 120 | 180 |
| 矿石 | 40 | 50 | 75 |
| 城门HP | 800 | 1000 | 1500 |
| 炉温 | 20℃ | 20℃ | 20℃ |

### 5.3 资源消耗机制

#### 食物消耗（每秒，`_decayResources` 每1s调用）

| 时段 | Easy | Normal | Hard |
|------|------|--------|------|
| 白天 | 0.7/s | 1.0/s | 1.5/s |
| 夜晚 | 1.05/s | 1.5/s | 2.25/s |

公式：`foodDecayDay = 1.0 × decayMult`，`foodDecayNight = 1.5 × decayMult`（decayMult：Easy=0.7 / Normal=1.0 / Hard=1.5）

#### 煤炭消耗（自动烧煤，有煤时）

每隔 `coalBurnTicks` ticks（1 tick = 200ms）消耗1单位煤炭：

| 难度 | coalBurnTicks | 等效速率 |
|------|--------------|---------|
| Easy | 10 ticks | 0.5个/s |
| Normal | 7 ticks | 0.71个/s |
| Hard | 5 ticks | 1.0个/s |

- **有煤**：自动烧煤，炉温不下降
- **无煤**：炉温按速率下降，且受暴风雪事件 `tempDecayMultiplier` 影响

#### 炉温衰减（无煤时，每秒）

| 时段 | Easy | Normal | Hard |
|------|------|--------|------|
| 白天 | -0.105℃/s | -0.15℃/s | -0.225℃/s |
| 夜晚 | -0.28℃/s | -0.4℃/s | -0.6℃/s |

暴风雪事件期间：`tempDecayMultiplier = 2.0`（持续60s），上述数值再×2

#### 矿石自动修城门

条件：`tickCounter % 10 === 0`（每2s）且 `ore > 0` 且 `gateHp < gateMaxHp`
- 消耗：1矿石
- 修复：`min(5, gateMaxHp - gateHp)` HP（不超过缺失量）

### 5.4 资源上限保护

所有来源的资源补充都受上限约束（代码中均有 `Math.min(上限, 当前+补充)` 保护）

## 6. 工作指令系统

### 6.1 指令总览

观众在直播间发送数字弹幕触发对应指令：

| 指令 | 弹幕 | 白天有效 | 夜晚有效 | 工作类型 |
|------|------|---------|---------|---------|
| 1 | "1" | ✓ | ✓ | 采集食物 |
| 2 | "2" | ✓ | ✓ | 挖掘煤炭 |
| 3 | "3" | ✓ | ✓ | 开采矿石 |
| 4 | "4" | ✓ | ✓ | 添柴升温 |
| 5 | — | — | — | **已从v2.0移除**（修城门指令，代码中彻底删除） |
| 6 | "6" | ✗ | ✓ | 攻击怪物（仅夜晚） |

> 注：服务器 `handleComment` 处理范围 `cmd >= 1 && cmd <= 4`，cmd=5 服务器端无处理逻辑

### 6.2 各指令产出（受效率倍率影响）

#### 指令1：采集食物
- 产出公式：`+Math.round(5 × totalMult × foodBonus)`
- foodBonus：丰收事件期间=1.5，其余=1.0
- 贡献值：+1

#### 指令2：挖掘煤炭
- 产出公式：`+Math.round(3 × totalMult)`
- 贡献值：+1

#### 指令3：开采矿石
- 产出公式：`+Math.round(2 × totalMult × oreBonus)`
- oreBonus：矿脉事件期间=2.0，其余=1.0
- 贡献值：+1

#### 指令4：添柴升温
- 炉温产出：`+Math.round(3 × totalMult)` ℃（也受效率倍率影响）
- 煤炭消耗：-1煤炭/次（即时扣除，与烧煤独立）
- 贡献值：+1

#### 指令6：攻击怪物（夜晚专属）
- **服务器端**攻击伤害：10点（固定，不受效率倍率影响）
- **客户端（Unity）本地伤害**：`WORKER_ATTACK_DAMAGE = 5`，攻击间隔 `ATTACK_INTERVAL = 1.2s`
  - 客户端 5 点/次为视觉反馈（TakeDamage 即时更新 HP 条）
  - 服务器的 10 点/次为权威数值，最终HP由服务器 `worker_hp_update` 快照决定
- 目标优先级：普通怪 → 精英怪 → Boss（从 `_activeMonsters` Map 中按类型优先）
- 贡献奖励（命中）：+5积分
- 贡献奖励（击杀）：普通怪+20 / 精英怪+50 / Boss+200（同时入积分池）
- Boss击杀 → 立即结束当夜（`_endNight()`）

### 6.3 效率倍率系统

```
totalMult = Min(3.0, playerBonus × globalBoost × broadcasterBoost × eff666Boost)
```

| 变量名 | 来源 | 值 | 范围 | 持续时间 |
|--------|------|-----|------|---------|
| `playerBonus` | 仙女棒累计 | `1.0 + _playerEfficiencyBonus[pid]` | 1.0 ~ 2.0（+5%/根，上限+100%） | 永久 |
| `globalBoost` | 能量电池 | `_playerTempBoost[pid]`（**仅发送者**） | 1.0 / 1.3 | 180s（可刷新） |
| `broadcasterBoost` | 主播加速 | `broadcasterEfficiencyMultiplier` | 1.0 / 2.0 | 30s（默认） |
| `eff666Boost` | 666弹幕/能力药丸 | `efficiency666Bonus` | 1.0 / 1.15 / 1.5 | 30s（可刷新） |
| **上限** | — | **3.0** | — | — |

> ⚠️ **重要**：`eff666Boost` 由 `efficiency666Bonus` 变量承载，**666弹幕和能力药丸共享同一变量和计时器**（见第7章礼物系统说明）

### 6.4 工位系统（WorkerManager）

- 每种指令（1-4, 6）对应4个工位（StationSlot，均匀分布在资源对象周围，圆形0°/90°/180°/270°）
- **工位动态重算**：运行时 `Awake()` 自动扫描场景对象坐标：

## 8. 怪物波次系统

### 8.1 怪物追踪与伤害机制（服务器端）

服务器维护两套并行系统：

**系统A：`_activeMonsters` Map（HP追踪）**
- 夜晚开始时（`_initActiveMonsters`）按配置创建所有怪物的HP记录
- 用于处理玩家攻击命中（指令6）和怪物AOE伤害（T5礼物）
- 类型：`normal` / `elite` / `boss`

**系统B：`_scheduleNightWaves`（波次调度）**
- 按 `beginning` 延迟后开始，每隔 `refreshTime` 发一次 `monster_wave` 消息给客户端
- 客户端收到后在场景中生成可见怪物
- 同时在 `_spawnWave` 中计算并执行伤害

**波次伤害分配（关键机制）**：
```javascript
const totalDamage = Math.floor(count × monsterGateDamage × workerDamageMult);
// 1. 均摊给所有存活矿工（至少1点）
const damagePerWorker = Math.max(1, Math.floor(totalDamage / aliveWorkers.length));
for (const [pid, w] of aliveWorkers) {
  w.hp -= Math.min(w.hp, damagePerWorker);
  if (w.hp <= 0) { w.isDead = true; ... }
}
// 2. 矿工全死后溢出伤害才打城门
if (remainingDamage > 0) this.gateHp -= remainingDamage;
```
> **核心设计**：矿工越多，城门越安全。矿工是城门的"肉盾"，全部阵亡后伤害溢出打城门。

### 8.2 刷怪方向（4种）

| spawnSide | 含义 | Unity场景坐标（参考） |
|-----------|------|---------------------|
| `left` | 左侧 | (-18, 0, 18) |
| `right` | 右侧 | (22, 0, 18) |
| `top` | 后方 | (5, 0, 28) |
| `all` | **随机单点**（见下方说明） | 等同于从left/right/top随机选一 |

每波随机选择其中一种：`sides = ['left', 'right', 'top', 'all']`

> ⚠️ **重要纠正（v1.2）**：`spawnSide = 'all'` **并非全方向同时刷新**。
> Unity `MonsterWaveSpawner.cs` 中 `switch(spawnSide)` 的 `default`（含 `case "all"`）直接调用 `RandomSpawnPoint()`，
> 即等效于随机选取 left/right/top 三点之一。因此 `sides` 数组中 'all' 比其他三个单点出现概率相同，
> 并**没有"从三个方向同时涌现"**的效果。

### 8.6 怪物AI（Unity MonsterController）

1. 目标优先级：`FindNearestAliveWorker()` → 无存活矿工时攻击城门（`_gateTarget`）
2. 当前目标死亡/逃跑 → 重新进入 Moving 状态重新寻目标
3. 攻击范围：4.0米 / 攻击间隔：1.5秒/次
4. **进入 Attacking 状态**：首次 `_attackCooldown = 0.5f`（等待0.5s后才开始首次攻击，非立即开打）
5. **客户端城门扣血已禁用**：`DoAttack()` 中 `CityGateSystem.TakeDamage()` 调用被注释掉；城门伤害完全由服务器 `resource_update` 权威推送，客户端只做动画播放
6. **受击白闪**：`ShowHitEffect()` → `MaterialPropertyBlock` 设置 `_BaseColor/_Color = white`，持续0.1s后清除（不创建材质实例）
7. 状态机：`Spawning`（枚举初始值，`Initialize()` 末尾立即切为 Moving）→ Moving → Attacking → Dead（死亡动画播放 `Sit` 即 Sitting Dazed）；枚举定义见 `MonsterController.cs:16`

### 8.7 同屏怪物上限（Unity端）

- `maxAliveMonsters = 15`：Unity场景中存活怪物不超过15只（超出排队等待）
- 注意：服务器端 `_activeMonsters` Map 追踪所有怪物（含排队中），与Unity的15只限制无关

- 血条更新立即生效（`fillAmount` 直接赋值 + Lerp平滑并行），避免受击时延迟显示
- 兜底重绑定：`_hpFillImage` 意外为 null 时 `RebindHpFillIfNeeded()` 自动重查找

### 13.0 本地排行系统（RankingSystem.cs）

**与服务器排行完全独立**，作为客户端本场数据的补充追踪：

| 功能 | 说明 |
|------|------|
| 生存贡献总分 | 工作指令+1/次，礼物按 contribution 值（由 SGM 事件驱动） |

> 注：服务器端贡献排行（live_ranking / survival_game_ended.rankings）是权威数据，RankingSystem 是本地快速查询的辅助缓存

### 13.1 实时贡献榜（live_ranking）

- **展示位置**：GameUIPanel 右侧（SurvivalLiveRankingUI，常驻不被SetActive关闭）
- **数量**：Top 5
- **更新**：贡献变化 → 防抖1.5s → `live_ranking` 广播
- **条件**：`state === 'day' || state === 'night'` 时才广播（其他状态跳过）
- **字段**：rank / playerId / playerName / contribution

### 13.2 本周贡献榜（weekly_ranking）

- **持久化文件**：`/opt/drscfz/data/weekly_ranking_{roomId}.json`（每个房间独立文件）
- **重置**：每周一00:00（UTC+8）—— 每次操作前 `_checkAndReset()` 自动检查周次
- **积分累计**：`addGameResult()` 每局结束将本局贡献**累加**到本周总分（非替换）
- **展示数量**：Top 50（请求时可指定，上限50）
- **触发**：每局结算后500ms自动广播；客户端 `get_weekly_ranking` 主动请求
- **payload字段**：`week`（"2026-W12" ISO周格式，UTC+8）/ `resetAt`（下周一UTC毫秒） / `rankings`
- **rankings字段**：rank / playerId / nickname / weeklyScore（累计值，整数）
- ⚠️ C# `WeeklyRankingEntry` 使用 `nickname` 字段（不是 `playerName`），排查显示问题时注意字段名

### 13.3 主播排行榜（streamer_ranking）

- **持久化文件**：`/opt/drscfz/data/streamer_ranking.json`（**全局单文件**，所有房间共享，不按房间隔离）
- **范围**：全平台共享（所有房间共用一个 `StreamerRankingStore` 单例）
- **排名依据**：difficulty_weight × maxDays（综合得分）
- **难度权重**：`DIFFICULTY_WEIGHT = { normal:1, hard:2, hell:3 }`
  - `hell` 权重已在服务器代码中定义但**未对外开放**（无 difficulty='hell' 入口）
- **展示数量**：Top 10
- **字段**：rank / streamerId / streamerName / maxDifficulty / maxDays / totalWins / totalGames / score
- **只记录最佳**：`addGameResult()` 仅在新 score > 旧 score 时更新（不是累计）

### 18.5a NetworkManager 连接参数

| 参数 | 值 | 说明 |
|------|-----|------|
| heartbeatInterval | 30s | 客户端发 heartbeat 间隔 |
| heartbeatTimeout | 45s | 未收到 heartbeat_ack 超时断线 |

| maxReconnectAttempts | 5 | 最大自动重连次数（GameConfig.cs 加载覆盖） |

> ⚠️ `NetworkManager.cs` Inspector 默认值为 3，但 `Awake()` 中会从 `Resources/Config/GameConfig` 加载并覆盖，`GameConfig.cs` 定义值为 5，**实际生效值为 5**（而非3）。

**连接流程**（严格顺序）：

**正常模式（直播伴侣启动）：**
1. 解析命令行参数 `-token=xxx`（支持 `-token=xxx` 或 `-token xxx` 两种格式）
2. POST `http://101.34.30.65:8081/api/douyin/init`（body: `{"token":"xxx"}`）→ 获取 roomId（JSON body方式，非查询参数）
3. WebSocket 连接 `ws://101.34.30.65:8081`
4. 发送 `join_room { roomId }` → 收到 `join_room_confirm`
5. 启动 ReceiveLoop + HeartbeatLoop

**GM模式（无token，本地开发）：**
1. 无命令行token → `IsGMMode = true`，roomId 直接设为 `"default"`
2. 跳过 douyin/init，直接 WebSocket 连接
3. 后续同上步骤4~5

> **线程安全**：WebSocket 接收在子线程，通过 `ConcurrentQueue` 在 Update() 主线程分发消息
> **WS接收缓冲**：8192 bytes；**HTTP init 超时**：15秒

### 18.5c 抖音推送数据格式（DouyinAPI.js）

**推送 body 格式**（抖音服务器 → 我们的 `/api/douyin/push`）：
```json
{
  "roomid": "直播间ID",
  "msg_type": "live_comment" | "live_gift" | "live_like" | "live_fansclub",
  "payload": "[{...}, {...}]"
}
```
> ⚠️ `payload` 是 **JSON 字符串**（非对象），服务器必须 `JSON.parse(payload)` 后才能访问字段。

**live_gift 单条 payload 字段**：
| 字段 | 类型 | 说明 |
|------|------|------|
| msg_id | string | 消息去重ID（5分钟TTL，每分钟清理） |
| sec_openid | string | 发送者匿名开放ID |
| sec_gift_id | string | 抖音礼物ID（正式上线后匹配用，现为TBD） |
| gift_num | number | 本次发送数量 |
| gift_value | number | **单价（分）**（SurvivalRoom.js将其作为单价，乘以gift_num得totalValue用于匹配） |
| nickname | string | 发送者昵称 |
| avatar_url | string | 头像URL |
| test | boolean | true=测试礼物，**DouyinAPI.js 已在回调前过滤** |

**live_like 字段**：`sec_openid, like_num`（like_num 为上游2秒聚合数据）

**live_comment 字段**：`sec_openid, nickname, content`（触发 survival_player_joined 首次评论）

**live_fansclub 字段**：`fansclub_reason_type (1=升级, 2=加团), fansclub_level`（游戏当前未处理此类消息）

**DouyinAPI 运行时参数**：
| 项目 | 值 | 说明 |
|------|-----|------|
| access_token TTL | 7200s（2小时） | 提前300s自动刷新 |
| Token API | `POST https://minigame.zijieapi.com/mgplatform/api/apps/v2/token` | appid+secret+grant_type=client_credential |
| 推送任务自动续约 | 每30分钟 | 防止任务过期（_renewAllTasks） |
| msg_id去重TTL | 5分钟 | 每分钟清理过期条目 |
| HTTP请求超时 | 10秒 | DouyinAPI自身对外发出的请求 |

**签名验证算法**（服务器侧验证推送来源）：
1. 取Headers：`x-nonce-str, x-timestamp, x-roomid, x-msg-type`（排除 x-signature 和 content-type）
2. 按key字典序排序，拼接为 `key1=value1&key2=value2&...`
3. 直接追加 rawBody原始字符串 + pushSecret（优先）或 appSecret（无连接符）
4. `MD5(拼接字符串) → Base64` = expected signature

### 18.5d 房间路由策略（RoomManager.js）

**抖音推送路由优先级**（`_getTargetRooms`）：
1. **精确匹配**：推送 roomId 对应的房间
2. **default 兜底**：`"default"` 房间有在线客户端时也转发（方便GM测试）
3. **回退策略**：若精确匹配的房间无客户端，转发到任意有客户端的活跃房间
   - 场景：主播中途关播重开 → 抖音分配新 roomId，但客户端仍连着旧 roomId
4. **创建新房间**：若以上都无匹配，自动创建新房间（保持兼容）

**房间生命周期**：
| 事件 | 行为 |
|------|------|
| 客户端连接 | `getOrCreateRoom(roomId)` |
| 客户端断线 | `removeClient(ws)`，房间进入 paused |
| 清理定时器 | 每 5 分钟扫描，paused 且 `lastActiveAt > 30分钟` 的房间销毁 |
| 推送到达 | 刷新 `lastActiveAt`（直播还在播=不销毁） |

### 18.6 关键架构决策

1. **服务器权威**：客户端等服务器 `survival_game_state` 或 `phase_changed` 确认才切换状态
2. **UI预创建**：所有UI面板 Editor脚本预建，运行时不 Instantiate
3. **资源产出服务器驱动**：客户端无本地定时器，所有资源变化来自服务器推送
4. **双重失败检测**：服务器+Unity客户端都有失败检测，以服务器 `survival_game_ended` 为最终结算信号
5. **工位动态绑定**：WorkerManager运行时扫描场景对象名自动重算工位坐标
6. **脚本挂Canvas（常驻）**：UI脚本挂Canvas always-active节点，禁止在Awake()中SetActive(false)

## 21. 城墙屏障系统

### 21.1 系统概述

`WallBarrier.cs`（静态类）：用纯代码实现城墙通行限制，不依赖 Unity Physics。

### 21.2 城墙参数（Running状态初始化）

```csharp
// SurvivalGameManager.ChangeState(Running) 时调用：
WallBarrier.Initialize(wallZ: -7.5f, gateMinX: -1.5f, gateMaxX: 3.5f, thickness: 2.0f);
```

> ⚠️ **Initialize() 调用前的静态字段默认值**（`WallBarrier.cs:22-26`）：
> `_gateMinX = -2.5f`、`_gateMaxX = 4.5f`、`_wallThickness = 1.5f`（与初始化后的值不同）。
> 若 `ClampMovement` / `GetDetourPoint` 在 `ChangeState(Running)` 之前被调用，将使用错误边界。

| 参数 | 值 | 说明 |
|------|-----|------|
| wallZ | -7.5f | 城墙 Z 轴防线（chengqiang-chengmen bounds 中心约 -7.35） |
| gateMinX | -1.5f | 城门缺口左边界 X |
| gateMaxX | 3.5f | 城门缺口右边界 X |
| thickness | 2.0f | 墙壁厚度（防止高速穿透） |
| 城门中心 | X=1.0 | `(gateMinX+gateMaxX)/2 = 1.0` |

### 21.3 通行逻辑

```
移动路径是否穿越 wallZ±thickness/2？
├─ 否（同侧） → 直接放行
├─ 是且 X 在 [gateMinX, gateMaxX] → 城门口，放行
└─ 是且 X 不在缺口 → 贴墙阻止：内侧者贴 wallZMin / 外侧者贴 wallZMax
```

### 21.4 绕行逻辑（GetDetourPoint）

Worker/Monster 从城墙一侧前往另一侧时：
- 当前 X 已在缺口内 → 直接穿行（无需绕行）
- 当前 X 不在缺口内 → 返回城门口中心 (1.0, 0, -7.5) 作为中间路径点，先走到缺口再穿过

### 30.4 持久化与衰减机制

#### 持久化

- 服务器维护 `_lifetimeContrib[playerId]`（终身累计贡献，持久化存储，**独立于**本局 `contributions[playerId]`）
- 本局 `contributions` 仍用于实时贡献榜、结算积分分配，保持现有逻辑完全不变
- `_lifetimeContrib` 随 `_addContribution()` 同步累加，持久化至存储（参考现有 `WeeklyRankingStore.js` 方案）
- 等级派生自 `_lifetimeContrib`，每次 `_addContribution()` 后调用 `_checkLevelUp(playerId)`

#### 每日衰减

| 条件 | 衰减公式 |
|------|---------|
| 当日未参与任何局 | `newLevel = floor(currentLevel × 0.95)` |
| 当日参与过至少1局（活跃减免） | `newLevel = floor(currentLevel × 0.975)` |

- 触发时机：UTC+8 自然日 00:00 结算，不按场次衰减
- 最低降至 Lv.1，皮肤/被动始终跟随当前等级对应阶段

#### 各类玩家自然稳定区间

| 观众类型 | 典型每场贡献 | 稳定等级区间 |
|---------|-----------|------------|
| 纯弹幕活跃 | 50–100 | Lv.20–30（阶3） |
| 偶尔小礼物 | 200–600 | Lv.35–50（阶4–5） |
| 定期T4/T5 | 2000–5000 | Lv.55–70（阶6–7） |

### 30.5 威胁值系统（MonsterController 目标选择）

怪物选择攻击目标时，使用威胁值加权：

```
目标得分 = 威胁值倍率 / √距离
```

怪物选择得分最高的存活矿工作为目标。**设计效果**：高等级矿工吸引更多怪物，成为团队"肉盾"；低等级矿工受到保护，安全输出。夜间防守位矿工聚集，距离相近时威胁值主导选择，正是最需要保护低等级玩家的时机。

**Boss特殊规则**：Boss 使用 **0.5× 威胁值效果**，防止Boss专盯高等级矿工或随机盯上新入场 Lv.1 矿工：

```
Boss目标得分 = (1 + 威胁值倍率 × 0.5) / √(max(距离, 0.1))
```

**MonsterController.cs 修改**：`FindNearestAliveWorker()` 改为按威胁得分排序，从 `WorkerManager._activeWorkers` 获取每个矿工的阶段（tier）信息。

### 30.6 动态难度补偿

当局所有参与过贡献的玩家平均等级较高时，怪物 HP 和数量略微增强，维持合作压力：

| 全场平均等级 | 怪物HP加成（Easy/Normal） | 怪物HP加成（Hard） | 怪物数量加成 | 攻击伤害 |
|------------|----------------------|-----------------|-----------|--------|
| < 20 | +0% | +0% | +0% | 不变 |
| 20–40 | +15% | +10% | +0% | 不变 |
| 40–60 | +30% | +20% | +10% | 不变 |
| > 60 | +50% | +25% | +20% | 不变 |

> **攻击伤害不参与动态调整**，仅提升怪物耐久和数量，避免低等级新入场玩家被秒杀。
>
> **与基础难度倍率的叠加方式**：动态难度加成应用于基础难度倍率之后（乘法叠加）。示例：Hard难度（×1.5）+ 平均等级>60（+25%）→ 最终 HP = 基础 × 1.5 × 1.25 = × **1.875**。Hard难度封顶+25%（而非+50%）即为此设计，防止 ×2.25 的极端情况。
>
> **随机Hard Night事件叠加上限**：当随机"Hard Night"事件与动态难度同时触发时，两者的HP加成取 max 而非相加，防止双重叠加失控。
>
> **平均等级统计范围**：统计当局所有参与过贡献的玩家（含超出WorkerPool上限12人之外的玩家），不限于场上可见的矿工。

### 30.7 皮肤选择

- 达到新阶段时**自动切换**为该阶段皮肤，触发阶段升级公告
- 玩家可通过弹幕**手动切换**已解锁的任意阶段皮肤：

#### 升级检查（_addContribution 末尾调用）

```javascript
_checkLevelUp(playerId) {
    const lc  = this._lifetimeContrib[playerId] || 0;
    const cur = this._playerLevel[playerId]     || 1;
    let newLevel = 1;
    for (let lv = 100; lv >= 1; lv--) {
        const tier = Math.floor((lv - 1) / 10);
        const lvInTier = ((lv - 1) % 10) + 1;
        const needed = TIER_THRESHOLDS[tier] + (lvInTier - 1) * TIER_COST_PER_LV[tier];
        if (lc >= needed) { newLevel = lv; break; }
    }
    if (newLevel <= cur) return;
    this._playerLevel[playerId] = newLevel;
    const newTier = Math.ceil(newLevel / 10);
    const oldTier = Math.ceil(cur / 10);
    if (newTier > oldTier) {
        this._playerSkinId[playerId] = TIER_SKINS[newTier - 1];
        this._broadcast('worker_level_up', {
            playerId,
            playerName: this._getPlayerName(playerId),
            newLevel, newTier,
            skinId: TIER_SKINS[newTier - 1]
        });
    }
}
```

WorkerController FindTargetWorker(bool isBoss) {
    float bestScore = -1f;
    WorkerController best = null;
    foreach (var wc in WorkerManager.Instance._activeWorkers) {
        if (!wc.IsAlive) continue;
        float threat = isBoss
            ? 1f + wc.ThreatMultiplier * 0.5f
            : wc.ThreatMultiplier;
        float dist = Vector3.Distance(transform.position, wc.transform.position);
        float score = threat / Mathf.Sqrt(Mathf.Max(dist, 0.1f));
        if (score > bestScore) { bestScore = score; best = wc; }
    }
    return best;
}

### 31.2 第一阶段：行为变种（🆕 待实现）

#### 概述

不引入新模型，仅修改目标选择逻辑，通过 `MaterialPropertyBlock` 色调区分。Day 3 起开始出现。

#### 变种定义

| variant | 名称 | 颜色 | 目标逻辑 | 数值 | 出现条件 |
|---------|------|------|---------|------|---------|
| `normal`（现有） | 标准怪 | 原色 | 攻击最近存活矿工 | 原配置 | 全程 |
| `rush` | 冲锋怪 | 红色色调 | **无视矿工，直奔城门** | HP ×0.5，速度 ×1.6 | Day 3 起，每波 1~2 只 |
| `assassin` | 刺客怪 | 深紫色调 | **优先攻击 HP 最低的存活矿工** | 与普通怪相同 | Day 4 起，每波 1 只 |

#### 冲锋怪设计意图

冲锋怪 HP 减半，击杀成本低，但绕过矿工直接攻城门。出现时团队必须有人专门截击（弹幕6指向城门区域），否则城门压力倍增。出现数量：

| 天数 | 每波冲锋怪数量 |
|------|-------------|
| Day 3–6 | 1 |
| Day 7–14 | 2 |
| Day 15+ | 3 |

#### 刺客怪设计意图

刺客怪专盯残血矿工，与矿工等级/威胁值无关，纯 HP 排序。高等级矿工如果负伤就会被绕过，低等级矿工复活后如果没被照顾到反而优先被锁定。促使团队关注队友状态。
